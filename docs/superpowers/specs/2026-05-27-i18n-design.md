# i18n 国际化设计文档

## 背景

fi-code 目前所有 UI 文本（Web 前端、TUI、后端错误消息）均为硬编码中文。为了支持国际化用户群体，需要引入完整的 i18n 体系，支持中文/英文，并预留扩展能力。

## 目标

1. **Web 前端**：所有用户可见文本支持中文/英文切换，自动检测浏览器语言。
2. **TUI 终端界面**：所有用户可见文本支持中文/英文切换，自动检测系统 `LANG` 环境变量。
3. **后端错误消息**：SSE 流中的错误消息根据用户语言偏好返回对应语言文本。
4. **预留扩展**：键名规范支持未来添加日语、法语等语言。

## 架构：方案 A（分别独立实现 + 统一键名规范）

Web（React）和 TUI/后端（Rust）技术栈完全不同，各自使用生态成熟的 i18n 库，通过**约定的键名规范**保证一致性。

## Web 前端（React）

### 技术选型

- `i18next` — 核心 i18n 框架
- `react-i18next` — React 绑定
- `i18next-browser-languagedetector` — 自动检测浏览器语言
- `i18next-localstorage-backend` — 持久化语言偏好到 localStorage

### 文件结构

```
frontend/src/i18n/
├── index.ts              # i18n 初始化：配置检测器、回退语言、资源加载
├── locales/
│   ├── zh.json           # 中文语言包
│   └── en.json           # 英文语言包
└── LanguageSwitcher.tsx  # 语言切换组件（Header 右上角下拉）
```

### 语言检测逻辑

1. **首次访问**：`i18next-browser-languagedetector` 按优先级检测：
   - `localStorage` 中保存的偏好
   - 浏览器 `navigator.language`
   - HTML `lang` 属性
   - 回退到 `en`
2. **保存偏好**：切换语言后写入 `localStorage`
3. **切换 UI**：Header 右上角显示当前语言图标（🇨🇳 / 🇬🇧），点击下拉切换

```tsx

```

## TUI（Rust）

### 技术选型

- `rust-i18n` — Rust 标准 i18n 库，支持 YAML/TOML/JSON

### 文件结构

```
crates/tui/src/i18n/
├── mod.rs
└── locales/
    ├── zh.toml
    └── en.toml
```

### 语言检测逻辑

1. **启动时**：读取 `LANG` 环境变量（如 `zh_CN.UTF-8` → `zh`）
2. **保存偏好**：通过 TUI 的 slash command `/lang zh` 或 `/lang en` 切换，写入配置文件
3. **切换 UI**：设置面板中添加语言选择下拉框

### 键名规范

与 Web 前端**完全一致**，使用同一套键名：

```toml
[permission]
toolLabel = "工具"
riskLabel = "风险等级"
approve = "确认执行"
reject = "拒绝"

[question]
answered = "已回答"
```

### 使用方式

```rust
rust_i18n::i18n!("locales");

let text = t!("permission.approve");
let formatted = t!("tool.writeHighRisk", name = "write");
```

## 后端（Rust）

### 技术选型

- `rust-i18n` — 与 TUI 共享同一套库和语言文件

### 文件结构

```
crates/core/src/i18n/
├── mod.rs
└── locales/
    ├── zh.toml
    └── en.toml
```

### 语言传递机制

Web 模式：
1. 前端在 HTTP Header `X-Lang` 中传递当前语言（`zh` / `en`）
2. 后端 `handle_chat_endpoint` 读取 `X-Lang`，存入 `AppState` 或请求上下文
3. SSE 流中的错误消息、系统通知使用对应语言文本

TUI 模式：
1. TUI 启动时通过环境变量或命令行参数传递语言
2. 后端读取 `LANG` 环境变量

### 需要国际化的后端文本

```rust
// permission.rs
"sudo privilege escalation detected" → t!("error.sudoDetected")
"rm -rf dangerous operation detected" → t!("error.rmrfDetected")
"tool bash requires user confirmation" → t!("tool.bashHighRisk")
"tool {name} may modify files, requires confirmation" → t!("tool.writeHighRisk", name = name)
"tool {name} auto-approved (low risk)" → t!("tool.autoApproved", name = name)

// tools/mod.rs
"Permission denied: user rejected" → t!("error.permissionDenied")
"Permission confirmation timeout (30s)" → t!("error.permissionTimeout")
"Question response timeout (60s)" → t!("error.questionTimeout")
"Question response channel closed" → t!("error.channelClosed")
```

## 迁移策略（分三阶段）

### 第一阶段：基础设施 + 高频组件

- Web：搭建 i18n 框架（i18next 初始化、LanguageSwitcher 组件）
- TUI：搭建 rust-i18n 框架
- 后端：搭建 rust-i18n 框架，支持 X-Lang Header
- 迁移文本：PermissionAsk、QuestionAsk、ChatPanel 空状态、错误消息

### 第二阶段：后端错误消息全面迁移

- 将 `permission.rs`、`tools/mod.rs` 中的所有硬编码错误/提示文本替换为 `t!()` 调用
- 前端补充对应的语言键

### 第三阶段：剩余 UI 文本迁移

- 迁移所有剩余的组件文本（LogPanel、Settings、LeftDrawer 等）
- 全面测试两种语言下的 UI 显示

---

## 错误处理

1. **缺失翻译键**：回退到英文（`fallbackLng: 'en'`），并在开发环境打印警告
2. **插值变量缺失**：保留原始模板文本（如 `tool {name}...`），不崩溃
3. **语言文件加载失败**：回退到英文

## 测试策略

1. **单元测试**：验证 `t()` / `t!()` 在两种语言下的返回结果
2. **E2E 测试**：切换语言后验证关键 UI 文本是否正确变化
3. **回归测试**：确保新增翻译键不会破坏现有功能
