# 重构记录 —— 2026-05-14

19:16 ~ 19:30（约 14 分钟）

## 模块

TUI 层拆分：`crates/core/src/tui/` → `crates/tui/src/`

## 重构动机

1. **解耦核心逻辑与前端**：`fi-code-core` 原本同时承载业务逻辑（Agent、工具、Provider、Session）和 TUI 渲染代码，导致编译时间较长，且 TUI 依赖 ratatui/crossterm 等重型图形库拖慢纯 CLI/Server 构建。
2. **打破循环依赖**：TUI 组件需要消费 `core` 中的 `SseEvent`、`Part`、`AppEvent` 等类型，而 `core` 中的工具模块（如 `ask_for_question`）又需要引用 `AppEvent`。将公共类型下沉到 `fi-code-shared` 后，双方都可以只依赖共享层。
3. **支持多前端独立演进**：CLI、Server、TUI、Desktop 各自有自己的二进制入口，TUI 独立成 crate 后，可以在不影响 core 的情况下迭代 UI。

## 具体改动

### 1. 新建 `crates/tui/` 独立 Crate

- 从 `core/src/tui/` 整体迁移以下文件到 `crates/tui/src/`：
  - `app.rs` —— 主应用循环、事件路由
  - `client.rs` —— HTTP 客户端封装
  - `components/` —— Chat、Input、LeftDrawer、RightDrawer、StatusBar、LogWindow、QuestionDialog、CardWidget、PartRenderer 等
  - `event.rs`（废弃，类型已迁移）
  - `layout.rs` —— 面板布局管理
  - `theme.rs` —— 主题系统
- 新增 `crates/tui/src/lib.rs`，导出 `run_tui_mode` 供 `fi-code-cli` 调用。
- `crates/tui/Cargo.toml` 依赖 `fi-code-core`、`fi-code-shared`、`ratatui`、`crossterm`、`tokio`、`reqwest` 等。

### 3. 更新入口与调用链

- `core/src/entry.rs`：`run()` 返回 `EntryOutcome`，默认模式不再直接启动 TUI，而是返回 `EntryOutcome::StartTui`，由 `fi-code-cli` 调用 `fi_code_tui::run_tui_mode(None)`。
- `crates/cli/src/main.rs`：根据 `EntryOutcome` 决定是否启动 TUI。
- `crates/core/src/lib.rs`：移除 `pub mod tui;`，新增 `pub mod tui_event;`（re-export）和 `pub mod theme_preset;`。

### 4. 修复编译问题

- `chat.rs` 中 `task.status` 原按 `TaskStatus` 枚举匹配，现改为字符串匹配（`String` vs `&str`），避免 `shared` 依赖 `tools` 模块。
- `app.rs` 中所有 `crate::tui::...` 路径更新为 `crate::...` 或 `fi_code_core::...`。
- `left_drawer.rs` 中 `FileNode` 统一使用 `fi_code_shared::dto::FileNode`。
- `status_bar.rs` 测试断言中 `LAT:2.4s` → `LAT: 2.4s` / `LAT:2.4s`（区分 standard/compact 模式）。
- 删除 `right_drawer.rs` 和 `left_drawer.rs` 的冗余 `FileNode` 本地定义。

### 5. 规避 rustc ICE（annotate_snippets）

rustc 1.94.1 在 `check_mod_deathness` / `annotate_snippets::renderer` 阶段会因特定 warning 触发 ICE（`slice index starts at X but ends at Y`）。通过清理 `fi-code-tui` 中所有 unused import / dead_code warning，成功规避该编译器 bug。

## 预期收益

1. **编译加速**：修改 TUI 代码时无需重新编译 `fi-code-core` 的全部业务逻辑。
2. **职责清晰**：`core` 专注业务逻辑，`tui` 专注终端渲染，`shared` 专注跨 crate 的 DTO/常量。
3. **测试隔离**：TUI 的 110 个单元测试可独立运行，不依赖 core 的完整初始化。
4. **未来扩展**：为 Desktop（Tauri）前端提供参考模式——独立 crate + shared DTO。
