# Bug List 2026-05-27

## 解决时间：13:38

### 模块：agent / tools / frontend

### 现象：
1. Web 模式下 `ask_for_question` 工具被调用时，其原始参数 JSON（包含 `question`、`options`、`recommended` 等字段）被直接展示在聊天记录中。
2. 问题弹窗（`PermissionDialog`）虽然触发，但可能因为原始 JSON 的干扰导致用户体验差。用户未在 60 秒内回答，触发超时错误。
3. `PermissionDialog` 使用 `absolute bottom-full` 定位，内容较多时向上延伸，顶部可能被浏览器视口截断。

### 根因：
1. **后端 `agent.rs` 和 `runner.rs`**：在 LLM 流式输出和工具执行前，会批量/实时发送 `SseEvent::Part { part: ToolUse { ... } }` 给前端。对于 `ask_for_question` 这种交互式工具，其参数 JSON 本就不应直接展示给用户，但后端没有过滤。
2. **前端 `ToolUsePart`**：直接渲染所有 `ToolUse` 的 `arguments` JSON，没有对 `ask_for_question` 做特殊处理。
3. **前端 `PermissionDialog` 定位**：`absolute bottom-full` 让弹窗底部对齐父容器顶部，内容多时向上生长，容易超出视口顶部被截断。

### 修复方案：
1. **后端跳过 `ask_for_question` 的 ToolUse SSE 发送**：
   - `agent.rs` line 571-588：在批量发送 `ToolUse` 事件时，如果 `name == "ask_for_question"`，直接 `continue` 跳过。
   - `runner.rs` line 193-199：在流式输出实时发送 `ToolUse` 时，用 `matches!` 过滤掉 `ask_for_question`。

2. **前端 `ToolUsePart` 兜底不渲染**：
   - `ToolUsePart.tsx`：如果 `part.name === 'ask_for_question'`，返回 `null`，避免任何情况下展示原始参数 JSON。

3. **前端 `PermissionDialog` 定位优化**：
   - 将 `absolute bottom-full` 改为 `fixed bottom-20`，确保弹窗始终停留在视口底部上方，不受容器滚动或截断影响。
   - 添加 `max-h-[60vh] overflow-y-auto`，防止内容过多时向上延伸超出视口。

### 相关文件：

- `crates/core/src/agent/runner.rs`
- `frontend/src/components/part-renderers/ToolUsePart.tsx`
- `frontend/src/components/PermissionDialog.tsx`

## 解决时间：15:49

### 模块：server/api/permission_api.rs, frontend/services/apiClient.ts

### 现象：
用户点击 `ask_for_question` 弹窗中的选项后，前端 `PermissionDialog` 抛出错误：
```
[PermissionDialog] Failed to respond question: Error: API returned no data
    at ApiClient.post (apiClient.ts:58:52)
```
导致用户回答无法送达后端，`wait_question_response` 最终因 60 秒超时而失败。

### 根因：
**前后端 API 响应语义不一致。**

1. **后端** `handle_question_respond` 和 `handle_permission_respond` 在操作成功后，返回 `ApiResponse::success(Value::Null)`，生成的 JSON 为：
   ```json
   { "success": true, "data": null, "error": null, "error_code": null }
   ```
   这在 Rust 语义中表示"操作成功，无返回数据"，是合理的 REST 设计。

2. **前端** `ApiClient.post<T>` 的校验逻辑过于严格（`apiClient.ts:58`）：
   ```typescript
   if (!data.success || data.data === null) throw new Error(data.error || 'API returned no data');
   ```
   该判断将 `data: null` 统一视为错误，即使 `success: true` 也无法通过。`serde_json::Value::Null` 序列化后即为 JSON `null`，因此 `data.data === null` 为 `true`，触发异常抛出。

### 修复方案：
**方案 A：后端返回非空标记对象（已采纳）。**

将 `ApiResponse::success(Value::Null)` 改为返回一个非空的 JSON 对象，确保 `data` 字段不为 `null`：

```rust
// permission_api.rs:102
Json(ApiResponse::success(serde_json::json!({ "received": true })))

// permission_api.rs:120
Json(ApiResponse::success(serde_json::json!({ "received": true })))
```

这样前端 `data.data === null` 为 `false`，校验通过，不再抛出错误。

**未采纳的方案 B**：前端放宽校验（`if (!data.success)` 不再检查 `data.data === null`），考虑到 `data: null` 在错误场景下也可能出现，统一放宽可能掩盖其他后端错误，因此选择由后端保证 `data` 非空。
