## 10:20 | TUI Chat 自动滚动 | 新消息到达后窗口不自动跟随到底部

**现象**：当 Chat 区域有新消息或 SSE 流式内容到达时，窗口不会自动滚动显示最新内容，用户需要手动按 PageDown 或滚轮滚动才能看到新内容。

**根因**：`Chat` 的 `scroll_offset` 是从**内容顶部**开始计数的，`scroll_offset = max_scroll_offset` 才表示底部。当用户已在底部时，新内容追加使 `max_scroll_offset` 增大，但 `scroll_offset` 仍保持旧的值不变，导致用户看到的是"旧的底部"，新内容被截断在下方不可见。

**修复方案**：
1. 新增 `auto_scroll: bool` 标志（默认 `true`）
2. `draw` 时若 `auto_scroll` 开启，直接使用当前计算出的 `max_scroll_offset` 作为视口起点，始终锁定底部
3. 用户手动向上滚动（`PageUp` / 滚轮向上）时暂停 `auto_scroll`
4. 用户回到底部时恢复 `auto_scroll`
5. `clear_messages` 时重置 `auto_scroll`
6. 补充 7 个单元测试覆盖各场景

## 10:26 | Tools Task Plan | 子任务执行 panic：timers are disabled

**现象**：执行 Task Plan（任务拆分）时，子任务执行阶段 panic，错误信息为 `A Tokio 1.x context was found, but timers are disabled`。

**根因**：`crates/core/src/tools/task/tool.rs` 中创建了一个 `tokio::runtime::Builder::new_current_thread()` 运行时来执行子任务，但**没有调用 `.enable_time()`** 启用计时器。而子任务内部调用 AI 客户端时，若遇到网络错误需要重试，`base_client.rs` 中的 `send_with_retry` 会使用 `tokio::time::sleep(backoff).await` 进行退避，导致 panic。

**修复方案**：在运行时构建器上添加 `.enable_time()`。

```rust
let rt = tokio::runtime::Builder::new_current_thread()
    .enable_time()
    .build()
    .unwrap();
```

## 17:40 | Agent SSE 流中断 | 对话过程中报错 "SSE stream interrupted while reading chunk"

**现象**：用户在对话中输入消息后，AI 回复过程中报错 `Agent loop error: SSE stream interrupted while reading chunk`，导致当前 Turn 完全失败。

**根因**：
1. **HTTP 客户端超时过短**：`provider.rs` 中创建 `reqwest::Client` 时硬编码了 180 秒总超时，但配置文件中默认超时为 300 秒。对于长对话或复杂生成任务，180 秒可能不足，导致 `reqwest` 在读取 SSE 流时主动断开连接。
2. **流读取错误无法恢复**：`send_with_retry` 只重试请求发送阶段的错误（连接失败、超时等）。一旦响应头返回成功，开始读取 SSE 流后，如果连接被意外关闭（网络抖动、API 服务器主动断开、负载均衡器重置等），`parse_openai_sse` 会直接返回 `Err`，`agent_loop` 随之失败，没有重试机会。

**修复方案**：
1. **使用配置中的超时值**：`Provider::new` 先读取 `Config` 中的 `timeout`（默认 300 秒），将其应用到 `reqwest::Client`，替代硬编码的 180 秒。
2. **增加 `run_one_turn` 级别的自动重试**：在 `agent_loop` 中，如果 `run_one_turn` 因为可重试错误（包含 "interrupted"、"timeout"、"connection"、"reset"、"broken pipe" 等关键词）失败，自动重试一次（最多 1 次）：
   - 重试前保存并恢复 `LoopState`（`messages`、`turn_count`、`token_usage`、`transition_reason`），确保状态干净
   - 重试前通过 `on_text` 回调向客户端发送通知 `"⚠️ 对话流中断，正在自动重试 (1/1)..."`
   - 重试前等待 2 秒，避免对 API 服务器造成瞬时压力
