# Bug List 2026-05-20

## 解决时间：20:52

### 模块：permission / tools / server / cli / agent

### 现象：
1. 权限系统 `PermissionChecker` 存在但从未被 `execute_tool_calls` 调用，所有工具（包括 `write`/`edit`/`bash`）在 Web/Server/TUI 模式下直接执行，没有任何权限确认。
2. CLI 模式下没有 `--dangerous` 参数，Ask 级别操作会阻塞等待 stdin 输入（但 stdin 在 server 模式下不可用）。
3. `ask_for_question` 工具在系统提示词中没有明确的使用条件，LLM 经常在应该询问用户时自行猜测。

### 根因：
1. `execute_tool_calls` 直接调用 `tool_call()`，跳过了 `PermissionAction::match_action()` 检查。
2. `PermissionChecker::check()` 只实现了 CLI stdin 交互，没有 SSE 事件通道来与前端/TUI 通信。
3. `prompt_template.md` 缺少 Interaction Rules 章节。

### 修复方案：
1. **权限系统集成到 `execute_tool_calls`**：
   - 在每个工具调用前插入 `PermissionAction::match_action()` 检查
   - CLI 模式（`on_tool_event` 为 `None`）：使用 `PermissionChecker::check_cli()`，默认拒绝 Ask 级别，`-d/--dangerous` 时通过
   - Web/Server/TUI 模式（`on_tool_event` 为 `Some`）：发送 `SseEvent::PermissionAsk` 事件，等待用户通过 `/api/permission/respond` API 响应

2. **全局权限响应通道**：
   - 在 `permission.rs` 中添加 `PERMISSION_RESPONSES: Mutex<HashMap<String, oneshot::Sender<bool>>>`
   - `wait_permission_response()` 创建 oneshot channel 并等待（30 秒超时）
   - `respond_permission()` 通过 API 接收用户确认并唤醒等待中的工具执行

3. **HTTP API `/api/permission/respond`**：
   - 新增 `permission_api.rs`，接收 `{ tool_call_id, approved }` 并调用 `respond_permission()`
   - 在 `server.rs` 路由中注册

4. **CLI `-d/--dangerous` 参数**：
   - `cli_args.rs` 新增 `dangerous: bool`
   - `entry.rs` 调用 `set_cli_dangerous(args.dangerous)`

5. **系统提示词更新**：
   - `prompt_template.md` 新增第 3 节 "Interaction Rules"
   - 明确 `ask_for_question` 的 4 种触发条件：路径歧义、意图不明确、破坏性操作、跨模块影响
   - 提供正反示例和选项设计原则

6. **测试适配**：
   - E2E/BDD 测试中 `collect_sse_events` 自动确认 `PermissionAsk`，避免测试阻塞
   - 单元测试 `test_execute_tool_calls_build_agent_allows_write` 开启 dangerous 模式
   - `test_prompt_builder_structure` 更新为检查新的第 3/4 节标题

## 解决时间：23:05

### 模块：tools / agent

### 现象：
系统提示词中已添加 Interaction Rules（第 3 节），明确告知 LLM 在路径歧义、意图不明确、破坏性操作、跨模块影响时应调用 `ask_for_question` 工具。但用户在实际使用 Web 模式时，`ask_for_question` 仍然没有被触发，LLM 继续自行猜测。

### 根因：
1. **工具 schema 描述太弱**：`ask_for_question` 的注册描述只有 `"Ask the user a question with predefined options"`，完全没有说明使用时机和触发条件。
2. LLM 决定是否调用工具时，**工具 `description` 是最关键的信号**。系统提示词中的规则说明虽然存在，但工具描述不明确导致 LLM 无法建立"遇到歧义 → 调用 ask_for_question"的强关联。
3. 对比同项目的 `handle_task_plan` 工具，其描述非常详细（明确说明"当你需要执行多个步骤时，必须使用此工具"），因此调用率很高。
4. `options` 数量限制为 1-3，与系统提示词中的"2-5 个选项"规则不一致。

### 修复方案：
1. **重写 `ask_for_question` 工具 schema 描述**：
   - 在 `description` 中完整列出 4 种触发条件（路径歧义、意图不明确、破坏性操作、跨模块影响）
   - 明确说明 "You MUST call this tool when... instead of guessing on your own"
   - 补充选项设计原则（2-5 个、标注推荐项、允许自定义）

2. **增强 schema 各字段的 description**：
   - `question`: "Be specific about the ambiguity or risk"
   - `recommended`: "The ID of the recommended option"
   - `allow_custom`: "Whether to allow the user to enter a custom answer"

3. **统一选项数量限制**：
   - 将 handler 中的 `options.len() > 3` 改为 `> 5`，与 schema `maxItems: 5` 和系统提示词规则保持一致

### 相关文件：
- `crates/core/src/tools/mod.rs` — 工具注册描述、schema、handler 限制

### 相关 Commit：
- 增强 ask_for_question 工具 schema 描述与触发条件
- 统一 options 数量限制 1-5

## 解决时间：23:45

### 模块：tools / compression / frontend

### 现象：
1. 使用 Web 模式读取大文件（如 `basic_tools.rs`，约 54K 字符）时，输出被严重截断，显示 `... [51710 chars truncated] ...`，LLM 只能看到 3,000 字符。
2. AI 在读取文件后停止生成，没有进行评审。
3. 提示词"读取文件并进行评审"没有触发 `handle_task_plan` 任务拆分。
4. 工具结果展示格式只有简单的 `✓ read: path (ms)`，没有行数/字符数等元信息。
5. read 结果在前端显示为 JSON 格式 `{"content":"..."}`，而不是代码块。

### 根因：
1. **双重截断叠加**：
   - 第一层：`OUTPUT_TRUNCATE_LENGTH = 50,000` — 工具层面硬截断
   - 第二层：`is_aggressive = true` 时 `READ_COMPRESS_THRESHOLD / 3 = 34,133` 字节触发 `compress_tool_result`
   - 第三层：`do_compress` 保留头部 1,000 + 尾部 2,000 = 3,000 字符，截断 47,000+ 字符
   - 虽然用户上下文充足（128K 限制，仅一条消息），但 `should_compress` 在对话历史较长时返回了 true

2. **AI 停止**：LLM 收到的文件内容被截断到 3,000 字符，无法完成评审，导致停止或输出无意义的重复内容。

3. **不拆分任务**：LLM 认为"读取+评审"是单步操作（读取文件后内容被截断，LLM 没有意识到还需要进一步分析）。

4. **展示格式简陋**：`execute_tool_calls` 中 `read` 的 `display_content` 只有路径和耗时，没有行数/字符数。

5. **JSON 展示**：截图中的 `{"content":"..."}` 是 LLM 生成的 `text` part（而非 `CodeBlock`）。LLM 在收到截断内容后，试图在自己的回复中重建/引用文件内容，输出为 JSON 格式。

### 修复方案：
1. **大幅提升截断限制**：
   - `OUTPUT_TRUNCATE_LENGTH`: 50,000 → 200,000（工具层面）
   - `READ_COMPRESS_THRESHOLD`: 100KB → 500KB（压缩层面）
   - `DEFAULT_COMPRESS_THRESHOLD`: 100KB → 500KB
   - `TOOL_RESULT_COMPRESS_HEAD`: 1,000 → 5,000
   - `TOOL_RESULT_COMPRESS_TAIL`: 2,000 → 10,000

2. **优化 read 工具展示格式**：
   - `display_content` 从 `✓ read: path (ms)` 改为 `✓ read: path | N lines | M chars (ms)`

3. **前端 ToolResultPart 增强**：
   - 显示 `content` 的第一行（元信息），让用户一眼看到文件规模

4. **同步更新压缩单元测试**：
   - 所有涉及阈值的测试用例调整为新的 500KB 阈值

### 相关文件：
- `crates/shared/src/constants.rs` — OUTPUT_TRUNCATE_LENGTH、COMPRESS_HEAD/TAIL
- `crates/core/src/agent/compression.rs` — READ/DEFAULT_COMPRESS_THRESHOLD、单元测试
- `crates/core/src/tools/mod.rs` — read 工具 display_content 格式
- `frontend/src/components/part-renderers/ToolResultPart.tsx` — 显示 content 第一行
